# NeonDB Performance Guide

NeonDB reducers run in one of three runtime tiers. Choose the tier that matches
your stage of development and throughput requirements.

---

## Runtime Tiers at a Glance

| Tier | Mechanism | Throughput (est.) | Use-case |
|------|-----------|-------------------|----------|
| 1 — JS | Boa interpreter (no JIT) | ~50 k calls/s | Rapid iteration, prototyping |
| 2 — WASM | Wasmtime + Cranelift JIT | ~500 k calls/s | Production, load tests |
| 3 — Native Rust | Compiled into server binary | ~2 M+ calls/s | Max-throughput hot paths |

Throughput figures are single-node estimates for simple read-modify-write
reducers on commodity hardware. Complex reducers (many table reads, large
fan-out) will be lower in all tiers.

---

## Tier 1 — JavaScript (Development)

NeonDB embeds [Boa](https://github.com/boa-dev/boa), a pure-Rust JS interpreter.
No V8, no Node, no native binaries — it works on every platform NeonDB supports.

**When to use:**
- Writing and testing new game logic
- Schema-exploration and prototyping
- Low-traffic internal tools

**Limitations:**
- No JIT — every call re-interprets bytecode
- Typical throughput ceiling: 50 k simple calls/s per core
- Do not use JS tier under real player load

**How it works:** Drop a `.js` file in `modules/`. The server loads it
automatically on start. No `neondb build` step required.

---

## Tier 2 — WASM (Production)

`neondb build` compiles every `.js` reducer in `modules/` to `.wasm` using
[Javy](https://github.com/bytecodealliance/javy) (QuickJS → WASM). The WASM
module is then executed by [Wasmtime](https://wasmtime.dev/) with the
[Cranelift](https://cranelift.dev/) JIT backend — a full native-code compiler.

**WASM-first auto-upgrade:** If a `.wasm` file exists alongside a `.js` file,
NeonDB automatically uses the WASM version. The log line
`"Using pre-compiled WASM for JS reducer"` confirms this is active.

**No code changes required** — the same JS source that runs in tier 1 becomes
a near-native WASM module. Zero rewrite cost.

**When to use:**
- Any time real players will be online
- Before running any load test or benchmark
- CI/CD pipelines after your logic is stable

**Quick upgrade:**
```bash
neondb build    # compile all .js → .wasm
neondb start    # server auto-picks .wasm over .js
```

Typical throughput: **~500 k calls/s** per core for a simple move reducer.

---

## Tier 3 — Native Rust (Maximum Throughput)

Native reducers are compiled directly INTO the NeonDB server binary. There is
no interpreter, no JIT warmup, and no cross-language ABI — just a Rust function
call from the worker thread.

**Function signature:**
```rust
fn my_reducer(ctx: &mut ReducerContext, args: &[u8]) -> neondb::Result<Vec<u8>> {
    // args is a MessagePack-encoded array
    // use ctx.get_row / ctx.set_row / ctx.stage_delta
    // return MessagePack-encoded result
}
```

**Registration** (in `src/reducer/registry.rs`):
```rust
registry.register_native("my_reducer", NativeReducerBackend::new(my_reducer));
```

**Requirements:**
- Clone the NeonDB source repository
- Add your reducer in `src/reducer/` or a new module
- Register it in `registry.rs`
- Recompile: `cargo build --release`
- Deploy your custom binary

**When to use:**
- A single reducer is your throughput bottleneck even after WASM
- You need deterministic sub-microsecond latency (e.g. physics tick, HFT-style auction)
- Your reducer logic is stable and unlikely to change often

Typical throughput: **~2 M+ calls/s** per core.

---

## Approximate Benchmarks

These numbers are from a single NeonDB node on a 4-core machine running a
read-modify-write player-move reducer. They are illustrative, not guarantees.

| Scenario | Tier | Calls/s |
|----------|------|---------|
| player move (JS) | Boa interpreter | 45 k |
| player move (WASM) | Wasmtime/Cranelift | 520 k |
| player move (native) | Rust direct call | 2.1 M |
| leaderboard scan 10 k rows (WASM) | Wasmtime | 18 k |
| leaderboard scan 10 k rows (native) | Rust | 210 k |

Fan-out (subscriptions) is additive: each extra subscriber adds ~1 µs of
`Arc<Bytes>` clone + channel send per call.

---

## When to Upgrade

```
Development ──> WASM before any load test ──> Native only if WASM isn't enough
```

Upgrade to WASM as soon as you move beyond local testing. Most games never
need native tier — 500 k calls/s with WASM is more than enough for tens of
thousands of concurrent players.

---

## Quick Commands

```bash
neondb build            # compile .js reducers → .wasm
neondb start            # start server (auto-selects .wasm when present)
neondb call <name> '[]' # invoke a reducer from the CLI
neondb watch "<query>"  # subscribe to a table
neondb seed seed.json   # bulk-load initial data (dev/test only)
```
