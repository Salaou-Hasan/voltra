# Project Structure

When you run `voltra init my-game --template voltra/basic`, you get a directory that looks like this:

```
my-game/
├── reducers.vol          <- YOUR GAME LOGIC. Edit this file.
├── voltra.toml            <- Server configuration
├── schema.toml            <- Optional: strict field validation
├── Cargo.toml             <- Rust package manifest
├── src/
│   ├── main.rs            <- Server bootstrap (rarely edit)
│   └── reducers.rs        <- AUTO-GENERATED by voltra build (never edit)
├── clients/
│   ├── VoltraClient.cs    <- Unity C# client SDK
│   ├── VoltraBehaviour.cs <- Unity MonoBehaviour helper
│   └── voltra_client.gd   <- Godot 4 GDScript client
└── data/
    └── wal/               <- Write-ahead log (created at runtime)
```

---

## reducers.vol

**This is the file you edit.** All of your game logic lives here.

It contains:
- `table` declarations — the data your game stores
- `reducer` functions — the actions clients can call

Example:

```voltra
table players {
    hp:   int  = 100,
    name: str  = "",
}

reducer heal(player_id: str, amount: int) {
    players[player_id].hp += amount
    return { ok: true }
}
```

Every time you change this file, run `voltra build` to recompile.

---

## src/reducers.rs

**Never edit this file.** It is automatically generated by `voltra build` and overwritten every time you build. Any changes you make here will be lost on the next build.

This file contains the native Rust code that `reducers.vol` compiles to. It is what makes Voltra fast — your game logic runs as compiled machine code, not interpreted script.

If you are curious, you can read it to understand what Voltra compiles to, but do not edit it.

---

## src/main.rs

The server bootstrap file. It starts Voltra, loads configuration, registers your reducers, and begins accepting WebSocket connections.

You will rarely need to edit this file. Common reasons to edit it:

- Adding a custom HTTP endpoint alongside the game server
- Changing how the server initializes (advanced use)
- Embedding third-party Rust libraries your reducers depend on

For most games, leave this file alone.

---

## Cargo.toml

The Rust package manifest. It defines your project name, version, and dependencies.

**Set your game name here:**

```toml
[package]
name = "my-game"        # change this to your game's name
version = "0.1.0"
edition = "2021"

[dependencies]
voltra = { path = "../Voltra" }  # or from crates.io
```

You may also add Rust dependencies here if your `src/main.rs` or any custom Rust code needs them. Your `.vol` reducers do not need Cargo.toml changes — the compiler handles that automatically.

---

## voltra.toml

Server configuration. Controls ports, WAL directory, authentication, schedulers, and more.

```toml
[server]
port = 3000              # WebSocket port clients connect to
metrics_port = 3001      # HTTP port for admin dashboard and healthz

[storage]
wal_dir = "./data/wal"   # where the write-ahead log is stored

[auth]
api_key = ""             # leave empty for open dev mode
                         # set a strong key for production

[schedulers]
# Run a reducer on a timer:
# cleanup = { reducer = "cleanup_sessions", interval_secs = 60 }
```

**Key settings:**

| Key | Default | Description |
|---|---|---|
| `server.port` | `3000` | WebSocket port for game clients |
| `server.metrics_port` | `3001` | HTTP port for admin and monitoring |
| `storage.wal_dir` | `./data/wal` | Where data is persisted to disk |
| `auth.api_key` | *(empty)* | Bearer token required from clients |
| `server.max_connections` | `10000` | Maximum simultaneous client connections |

## schema.toml

Optional strict validation. Without it, any JSON object can be stored in any table. With it, Voltra rejects rows that don't match the declared types.

```toml
[players]
fields = [
    { name = "hp",    type = "int",   required = true  },
    { name = "name",  type = "str",   required = true  },
    { name = "alive", type = "bool",  required = false },
]
```

Use `schema.toml` when:
- You want the server to enforce data integrity
- You are working in a team and want to prevent accidental bad writes
- You are building a production game and want to catch bugs early

The table declarations in `reducers.vol` are for documentation and editor tooling. `schema.toml` is what the server actually enforces at runtime.

## clients/

Ready-to-use client SDKs for connecting your game engine to Voltra.

### VoltraClient.cs + VoltraBehaviour.cs (Unity)

Drop these two files into your Unity project's `Assets/` folder. No additional packages required.

```csharp
// In your Unity scene:
public class GameManager : MonoBehaviour {
    private VoltraClient db;

    async void Start() {
        db = new VoltraClient("ws://127.0.0.1:3000");
        await db.Connect("my-api-key");
        await db.Call("spawn", new object[] { "player1", "Alice", 0f, 0f });
    }
}
```

### voltra_client.gd (Godot 4)

Add this as an Autoload in your Godot project (Project → Project Settings → Autoload).

```gdscript
# In any Godot node:
func _ready():
    Voltra.connect_to_server("ws://127.0.0.1:3000", "my-api-key")
    await Voltra.call_reducer("spawn", ["player1", "Alice", 0.0, 0.0])
```

---

## data/wal/

Created automatically when the server first starts. Contains the write-ahead log (WAL) — a durable record of every write. If the server crashes and restarts, it replays this log to restore state.

Do not delete this directory unless you want to wipe all game data. Back it up regularly for production deployments.

---

## The Build Flow

```
You edit:          reducers.vol
You run:           voltra build
Voltra generates:  src/reducers.rs
Rust compiles:     target/release/my-game.exe
You run:           voltra start
```

The only file in this entire project that you regularly edit is `reducers.vol`. Everything else is either generated, configuration, or rarely touched.
